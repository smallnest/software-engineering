Problem. SnakeGame class lives in two files — snake-game.js (ES module) and inline in snake.html (copy-pasted). Any logic change must be made in both places or they diverge.

Solution. Serve snake.html via local HTTP server so the browser can load snake-game.js as a module. Remove the inline copy. Falls back gracefully: add a <script src="snake-game.js"> with a UMD-style wrapper, or just document python3 -m http.server as the launch command.

  • • leverage: one interface, two call sites (renderer + tests)
  • • locality: logic change = one edit, not two
  • • deletion test: remove snake-game.js and tests break; remove inline copy and HTML breaks

2. Inject randomness seam into SnakGame

Worth exploring in-process

snake-game.js · snake-game.test.js

Before — hardcoded Math.random()

_placeFood()
Math.random() → food position
Tests can’t seed food placement.
Every food-placement assertion is best-effort.

After — injectable random function

constructor(cols, rows, random?)
this.random = random ?? Math.random
_placeFood() uses this.random()
Tests inject seeded PRNG → deterministic.

Problem. _placeFood() calls Math.random() directly. No seam exists to control food placement. The “eating food” test manually injects a food coordinate to work around this.

Solution. Accept optional random: () => number in the constructor, defaulting to Math.random. Tests can inject a seeded PRNG. Interface grows by one optional parameter; implementation doesn’t change.

  • • leverage: same interface, one optional seam for testability
  • • locality: randomness decision lives at construction, not inside _placeFood
  • • one adapter (Math.random) is the default; seeded PRNG is the second, justifying the seam

Problem. getState() returns the live this.snake array. A caller doing game.getState().snake.pop() corrupts internal game state. The interface leaks mutability without declaring it.

Solution. Return a shallow copy: snake: [...this.snake]. One line change. Callers get a snapshot; internal array stays protected behind the interface.

  • • leverage: interface contract strengthened — read-only snapshot
  • • locality: mutation protection lives at the seam, not in callers
  • • cost: one array allocation per getState() call (~20 elements max)

Top recommendation

Tackle Candidate 1 first — the SnakeGame duplication.

It’s the only candidate where the deletion test fails in both directions: delete either copy and something breaks. The other two candidates are interface refinements on a single module; they become easier to apply once there’s only one module to refine.

Candidates 2 and 3 are small, independent, and safe to apply after Candidate 1. They don’t block each other.
